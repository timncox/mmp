mmp.chat Landing Page — Which approach?

The landing page serves two audiences: end users ("how do I use this?") and developers ("how do I build with this?")

A

Single Page — Everything Inline

One scrollable page: hero → setup steps → how it works → protocol docs → footer. No navigation, no framework. Just a beautiful HTML file served by the existing Express server. Deploys with the MMP server itself.

Pros

  • Zero extra infrastructure
  • Ships in the same container
  • Fast to build (single HTML file)
  • Works offline

Cons

  • Protocol docs get long inline
  • No SEO-friendly sub-pages
B

Multi-Page — Separate Docs Site

A small Next.js or Astro site deployed alongside the MMP server. Landing page at /, docs at /docs, protocol spec at /spec. Proper navigation, search, SEO.

Pros

  • Clean separation of marketing vs docs
  • SEO-friendly URLs (/docs/encryption, /docs/tools)
  • Easier to maintain long-term

Cons

  • Separate deploy (or complex Docker setup)
  • More to build and maintain
  • Overkill for v1
C

Hybrid — Rich Landing + Spec Link

A polished single-page landing (hero, setup, how it works) served by the Express server. For protocol docs, link to the GitHub-rendered MMP-SPEC.md or a simple /spec route that renders the markdown. Best of both worlds.

Pros

  • Beautiful landing page, zero extra infra
  • Spec stays in markdown (easy to maintain)
  • Ships in the same container
  • Can upgrade to full docs site later

Cons

  • Spec rendering is basic (just markdown)

My recommendation: C — Hybrid. Polished landing page for users, spec rendered from markdown for developers. Ships in the same container, upgradeable later.
